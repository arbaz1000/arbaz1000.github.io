Module 1

Tender Summary

Index-then-extract: classify the document structure first, then extract fields scoped to relevant sections. Works across any client’s summary format without code changes.

The Problem

Tenders arrive as 10-30 documents spanning hundreds of pages. The Go/No-Go decision requires extracting eligibility, scope, timelines, and commercial terms scattered across the entire bundle. Days per tender, and incomplete reads risk disqualification. Every client uses a different summary format.

The Design

Two-phase pipeline: index the document structure, then extract fields scoped to relevant sections. The index step makes extraction efficient — each field only sees the pages that could contain it, not the full 300-page bundle.

Index the document structure, then extract scoped to relevant sections — every field is traceable
Phase 1: Classify pages into typed spans (19 categories). One LLM call per file, cached by hash.

Phase 2: Each template field has designated source categories (e.g., “Payment Terms” looks in SCC then GCC). Only relevant spans sent to LLM. All categories extracted in parallel. Every field carries a source citation.

Cross-module: The document map feeds downstream — TDS and approved vendor list spans pass to BOQ-to-RFQ, so each product gets the right spec sheet without re-reading the tender.

Key Decisions

Decision Why
Span classification as index Sending 300 pages to the LLM for each field is expensive and hallucinates. Building a document map first makes extraction scoped — each field only sees pages that could contain it.
Category-scoped extraction Once the index exists, each category is semantically independent — SOW, GCC, BOQ can be extracted in parallel with no coordination. Reduces both cost and latency.
Dynamic templates Each client has their own summary format with different fields and sections. Client uploads an Excel template, system generates the extraction schema. New client onboarding = one Excel upload.
Signature-based caching Results cached by hash of file set + template + model version. Same documents + same template = instant result, even across different tenders. Classification and extraction each cached independently — adding a new file only reclassifies that file.

Results

~5 min per tender — down from days of manual review across 10-30 documents
  • In use by PSU and EPC clients on the Workwise platform
Module 2

BOQ-to-RFQ

BOQ → vendor-ready RFQ packages with matched products, spec sheets, and approved makes. The hard part isn’t parsing — it’s interpreting what items mean, which traditionally bottlenecks on experienced engineers.

The Problem

BOQ line items bundle products with services, inherit descriptions from parent rows, and reference standards inline. An experienced engineer reads this structure and knows what inherits from what, which parts are products vs services, which accessories are associated items vs the main product. That interpretation lives in people’s heads and bottlenecks the entire procurement chain.

The Design

Five-stage pipeline: chunk → interpret → extract → match → link. The key design choice is separating interpretation from extraction — the LLM reasons about what items mean first, then a second pass extracts clean structured data.

V1: Fixed 30-row chunks, sequential, rolling context forward. Worked but didn’t adapt to content density — dense cable schedules got truncated, sparse sections wasted tokens.

V2: Semantic chunking (LLM identifies package boundaries from document structure) + scratchpad interpretation (reason about ambiguous items in free text before extracting structured data). Packages are self-contained → parallel processing. Extracts full BOQ hierarchy: packages → product groups → line items, with associated products and services separated.
V2 adds semantic chunking + scratchpad interpretation; V1 below for comparison

Key Decisions

Decision Why
Scratchpad before extraction Asking the LLM to interpret and extract in one pass conflates reasoning with structured output. A separate interpretation step lets the model reason about ambiguous descriptions in free-form text (higher temperature), then a second pass extracts clean structured data from that interpretation (lower temperature, strict schema). The temperature gradient matches the task — exploratory reasoning first, deterministic structuring second.
Semantic chunking (V1 → V2) V1’s fixed chunks worked — rolling context (forwarding the last product block) handled boundary splits. But chunk size didn’t adapt to content density, and sequential processing couldn’t scale. Semantic boundaries let the LLM split based on the document’s own structure, and since packages are self-contained, they can be processed in parallel.
Embed + LLM reranking Vector similarity against a 7k+ product catalog provides recall. An LLM reranking step applies domain-specific constraints — material must match, size must align — achieving precision without manual mapping rules. Matched products carry their vetted vendors from the database.
Semantic doc linking TDS spans from the tender summary index are classified into per-product sections. Each BOQ product is semantically matched to the right TDS section and approved makes entry. The LLM receives compact indices, not raw documents.

Results

8–10x faster — from BOQ upload to RFQ-ready output with matched vendors, specs, and approved makes
  • Handles BOQs with 1000+ line items across multiple sheets
  • In use by PSU and EPC clients on the Workwise platform
Module 3

Quotation Parsing

After vendor inquiries go out, quotes come back as PDFs — every vendor uses a different layout, different line item descriptions, different ways of bundling products. The pipeline extracts structured data from any quote format, then semantically matches it against the original RFQ to flag matches, deviations, and missing items.

What Made It Hard

  • No two vendors format quotes the same way — simple price tables, multi-page proposals with cover letters and T&Cs, panel BOMs with sub-components, ERP-generated exports. The parser handles 6+ distinct formats without per-vendor configuration
  • Multi-page table continuation — price tables span 3-4 pages with sequential row numbers. Per-page extraction must detect continuation (first row ≠ 1) and merge correctly
  • String matching fails for RFQ comparison — quote says “300 SQMM x 3C”, RFQ says “3.5C x 300 Sq.mm”. These are equivalent, but any distance metric misses it. Semantic matching via LLM is required

The Design

Two-stage pipeline: VLM extraction (per-page, parallelized, with multi-page fallback) then LLM semantic matching against RFQ.

Per-page VLM extraction → assembly → semantic RFQ matching → per-product comparison
Why VLM over OCR? Vendor quotes have wildly inconsistent layouts — merged cells, rotated headers, embedded images, Indian number formats (“8,56,930.00”). A vision language model reads the page as an image, the same way a human does, sidestepping layout parsing entirely. Per-page extraction runs in parallel (8 concurrent); if all pages yield zero items (e.g., specs on page 1, prices on page 2), a multi-page fallback sends all pages in one call.

Post-extraction enrichment runs before matching: make/brand propagation (if stated once globally, applied to all items), BOM component linking (panels list sub-components separately), supporting document matching (datasheets linked to parent items), and price validation (flags unit_price × qty ≠ total without auto-correcting — we don’t know which digit was misread).

RFQ matching uses LLM semantic comparison, not string distance. For each RFQ item, the LLM determines: exact match (product, size, quantity align), deviation (similar product but spec differs), or no match (vendor didn’t quote this item). Extra quoted items not in the RFQ are surfaced separately.

Key Decisions

Decision Why
VLM on page images Handles any PDF layout without OCR preprocessing. Indian number formats, special characters, and merged cells are read naturally from the visual page.
Per-page + fallback Per-page is fast and cheap (smaller images, parallel). But specs and prices sometimes live on different pages — zero-item result triggers automatic multi-page retry.
No auto-correction Price validation flags mismatches but never silently corrects — we can’t know which digit the VLM misread. Human review resolves flagged items.
Structured output (Pydantic) 49 fields per line item enforced via schema. Eliminates JSON parsing failures and ensures every downstream consumer gets clean, typed data.

Results

  • Parses 6+ distinct quote formats — simple tables, multi-page proposals, panel BOMs, multi-file submissions, ERP exports — without per-vendor configuration
  • Semantic RFQ matching catches equivalences that string matching misses — procurement teams see a structured comparison per vendor instead of manually cross-referencing PDFs
  • Full audit trail: every VLM/LLM call logged with token usage and cost estimates
