Module 1

Technical Bid Evaluation

The core technical challenge isn’t extraction or model orchestration — it’s encoding domain knowledge into system behavior that a domain expert would trust. What a stamp means contractually, which content is the bidder’s vs the purchaser’s, when “no annotation” is the answer. The engineering decisions — hybrid routing, prompt calibration, confidence tiers — all flow from getting the domain right first.

The Problem

In EPC procurement, a buyer sends technical specs to bidders. Each bidder responds by stamping the buyer’s own documents (datasheets, checklists, spec covers) and returning them alongside their own quotation letter. The result is a single PDF — 50-100+ pages — where most of the content is the buyer’s original text, and the bidder’s actual response is scattered across stamps, handwritten annotations, filled-in fields, and qualifying statements buried in fine print.

An engineer then checks this against an evaluation template (TBE), point by point, across 50-100+ evaluation points. 4-6 hours per bidder × 4-5 bidders per tender.

What Made It Hard

Most of the PDF is not the bidder’s content. ~90% of a bidder’s response is the buyer’s own documents stamped and returned unchanged. The printed text is the purchaser’s specs. The only signal is what the bidder added — stamps, handwritten annotations, filled-in fields, struck-through values, color ink marks. A naive “extract all text” approach drowns in purchaser content and misses the actual bidder response.

Stamps mean compliance. In EPC convention, a bidder stamping a spec cover page is a contractual declaration: “we accept this entire document.” A stamped page with zero modifications is the normal case — and it means full compliance. Missing this domain convention means wrong compliance status on every evaluation point that references that document.

Two failure modes pull in opposite directions. Let the model infer too freely and it fabricates evidence (“API 6D referenced in datasheet” — it never saw the datasheet). Restrict it too much and it refuses to credit valid compliance signals — stamped documents that contractually mean full acceptance get marked INSUFFICIENT_SUBMISSION. The prompt had to encode the exact boundary: accept blanket compliance from stamps, but state the basis honestly.

Every client uses a different template. Different groupings, categories, evaluation criteria. Hardcoding per client doesn’t scale.

The Design

Three-layer pipeline: schema understanding → bidder PDF parsing → group evaluation. Domain-agnostic — works across product categories without hardcoded schemas.

Each layer’s output feeds the next — schema + submission map → evaluation, line items + quotation → pricing
Step 0 solves the template problem: the system reads any client’s TBE Excel and uses an LLM to understand the structure — what the evaluation points are, how they’re grouped, which ones are technical vs commercial vs line items. The line items list is extracted here for downstream price verification. Runs once per template, cached by hash, reused across all bidders.

Layer 1 solves the bidder-vs-purchaser problem: for each page, it identifies what type of document this is (quotation, datasheet, stamped spec cover — 11 types), whether the bidder stamped it, and crucially, what the bidder actually wrote or changed on this page. Contiguous pages of the same type merge into a structured submission map — what’s in it, where, and what the bidder flagged or changed. The quotation and pricing pages identified here are extracted for downstream price verification alongside Step 0’s line items.

Layer 2 connects the two: it takes the grouped evaluation points from Step 0, matches them against Layer 1’s submission map, and evaluates each one — does this bidder’s submission address it? A stamped spec cover means blanket compliance. A flagged observation on the quotation means a qualification that might affect multiple evaluation points. Each evaluation point gets a status, confidence level, evidence pages, and flagged observations.

Key Decisions

Decision Why
Hybrid text/VLM routing VLM can’t read wide tables (pricing columns compressed to pixels). Text extraction can’t see stamps or handwriting. Per-page routing based on extractable text length — each page gets the right model.
Blanket compliance from stamps EPC convention: stamped spec cover = acceptance of entire document. Without this, every evaluation point referencing a stamped doc gets INSUFFICIENT_SUBMISSION. Domain knowledge encoded in the evaluation logic, not just extraction.
Confidence tiers HIGH = direct evidence, MEDIUM = inferred from blanket stamp, LOW = judgment call. Human reviewers check LOW first, skip HIGH — focuses review time on the rows that matter.
LLM-based TBE parsing Different clients have different TBE layouts. Generic LLM parser handles any format — new client onboarding is an Excel upload, not engineering work.
Three-level cache Schema cached per template hash (reused across bidders). PDF parsing cached per file hash. Full results cached per template + PDF combination. Same inputs = zero LLM calls.

Results

~3 min per bidder — down from 4-6 hours manual (90 evaluation points, 62-page PDFs)
  • Consistent, auditable output — every evaluation point has compliance status, confidence level, evidence page references, and flagged observations
  • Handles everything from bare-minimum quotation-only submissions to full technical packages with datasheets, certificates, and compliance forms — same pipeline, no per-bidder tuning
  • In use by PSU and EPC clients on the Workwise platform
Module 2

Financial & Credential Evaluation

A structured extraction pipeline that works across any client’s evaluation template without engineering work per client, produces auditable per-cell output with source citations, and handles the messy reality of Indian financial documents. The value is in the genericity and auditability, not in some deep technical challenge.

The Problem

Before awarding an EPC contract, the buyer checks each vendor’s financial and credential eligibility: annual turnover, net worth, years of experience, similar project history, certifications. An evaluator hunts through 5-15 documents per vendor, cross-references numbers against the tender’s evaluation template (an Excel sheet with fields, thresholds, and document requirements), and cites sources. 2-3 hours per vendor × 10-15 vendors per tender.

The core engineering problem is that no single document has all the fields, and the same field can appear in multiple documents with different representations — “23.47 Crores” in a CA certificate vs “23,47,18,942” in a balance sheet vs “approximately 25 Crores” in a self-declaration. The system maps each field to its designated document types, extracts from those, normalizes Indian financial notation, and resolves conflicts using a source authority hierarchy — which document type to trust over which, per field — encoding the same judgment an experienced evaluator applies manually.

The Design

Two-phase pipeline: understand the template + classify documents, then extract fields from their designated source types and resolve conflicts by source authority.

Extract from all sources, normalize, then resolve by document authority — every field is traceable
Phase 1 runs in parallel: the schema builder reads the client’s evaluation template and generates field specifications (what to extract, data types, thresholds), while the classifier categorizes each vendor document by type — balance sheet, F10, compliance certificate, self-declaration — and routes each to text or vision extraction based on whether it’s digital or scanned.

Phase 2 extracts each field from its designated document types — turnover from the F10 and balance sheet, certifications from scanned certificates, experience from self-declarations. Values are normalized (crore/lakh representations to raw INR) and merged: when the same field appears in multiple sources, the higher-authority source wins. Conflicts are surfaced with source citations — the evaluator sees the final value and any discrepancies, not a black-box answer.

Key Decisions

Decision Why
Extract-all, resolve later Each field has designated source types — turnover from the F10/balance sheet, experience from self-declarations, certifications from scanned certificates. Schema defines which document types to look in per field, and authority ranking picks the winner when multiple sources have the same value.
Source authority hierarchy CA-certified F10 outranks balance sheet outranks self-declaration for financial fields. For non-financial fields, the hierarchy shifts. Domain-specific ranking per field type, not a single global order.
LLM-based template parsing Each client has their own evaluation format. New client onboarding = one Excel upload, not engineering work.
Per-cell traceability Every extracted value carries its source document, page number, and confidence score. When sources disagree, the conflict is surfaced alongside the chosen value. Evaluators can verify and sign off.

Results

~4 min per vendor — 4-5x faster than manual evaluation across 5-15 documents per vendor
  • In use by PSU and EPC clients on the Workwise platform
