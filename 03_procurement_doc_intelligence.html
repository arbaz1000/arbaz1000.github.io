Module 1

Tender Summary

Index-then-extract — analogous to how a code indexer parses a codebase into symbols before querying, except tender documents have no formal grammar, so the indexing step itself is AI-powered. Works across any client’s summary format without code changes, with auditable per-field output.

The Problem

Tenders arrive as bundles of 10-30 documents — scope of work, general conditions, BOQs, technical specifications, annexures, corrigenda — spanning hundreds of pages across PDFs, Word docs, and spreadsheets. Before a procurement team can decide whether to bid, they need to assess: do we meet the eligibility criteria? Can we post the EMD? Is the deadline feasible? Does the scope match our capabilities? This Go/No-Go decision requires extracting and cross-referencing information scattered across the entire bundle. Days per tender, and incomplete reads risk disqualification or wasted bid effort.

Every client organization uses a different summary format (an Excel template with their own fields and sections). The system needs to adapt to any template without code changes.

The Design

Two-phase pipeline: index the document structure, then extract fields scoped to relevant sections. The index step makes extraction efficient — each field only sees the pages that could contain it, not the full 300-page bundle.

Index the document structure, then extract scoped to relevant sections — every field is traceable
Phase 1 builds a document map: each file’s pages are classified into typed spans — SOW pages 1-45, GCC pages 46-80, Technical Specs pages 81-120, and so on (19 document categories). This is the “indexing” step — one LLM call per file, cached by file hash.

Phase 2 uses the map: each field in the client’s template has designated source categories (e.g., “Payment Terms” looks in SCC then GCC). For each category, only the relevant spans are sent to the LLM with the fields that belong to that category — not the full document bundle. All categories run in parallel, so wall-clock time scales with the largest section, not the total page count. Every extracted field carries a source citation (e.g., “SOW | PAGE 5”) back to the originating document.

Cross-module output: The document map isn’t just for summarization. The technical specification and approved vendor list spans identified here are passed downstream to the BOQ-to-RFQ module, where they’re matched to individual line items — so each product in the BOQ gets the right spec sheet and approved makes without re-reading the tender documents.

Key Decisions

Decision Why
Span classification as index Sending 300 pages to the LLM for each field is expensive and hallucinates. Building a document map first makes extraction scoped — each field only sees pages that could contain it.
Category-scoped extraction Once the index exists, each category is semantically independent — SOW, GCC, BOQ can be extracted in parallel with no coordination. Reduces both cost and latency.
Dynamic templates Each client has their own summary format with different fields and sections. Client uploads an Excel template, system generates the extraction schema. New client onboarding = one Excel upload.
Signature-based caching Results cached by hash of file set + template + model version. Same documents + same template = instant result, even across different tenders. Classification and extraction each cached independently — adding a new file only reclassifies that file.

Results

~5 min per tender — down from days of manual review across 10-30 documents
  • In use by PSU and EPC clients on the Workwise platform
Module 2

BOQ-to-RFQ

Takes a Bill of Quantities (BOQ) — the schedule of items a buyer needs priced — and produces a complete Request for Quotation (RFQ) package per line item: structured product, matched vendor, attached spec sheet, approved makes. The interesting design problem is that BOQ interpretation requires domain expertise that traditionally bottlenecks on experienced engineers.

The Problem

BOQs arrive in inconsistent formats — PDFs, Excel files, scanned documents — with merged cells, multi-row headers, and no standard column structure. But the harder problem isn’t parsing the format. It’s interpreting what the items mean. A section might have a parent row “Supply and installation of lighting fittings complete with ballast, driver, diffuser, reflector, as required” followed by variant rows for different fixture types — each inheriting the parent description, bundling products with installation services, and referencing standards and approved makes inline. An experienced engineer reads this structure and understands what inherits from what, which parts are products versus services, and which accessories are associated items versus the main product. That interpretation currently lives in people’s heads, and it’s the bottleneck for the entire procurement chain.

Once items are interpreted and structured, each one needs to be matched to vetted vendors from the database, cross-referenced against the tender’s technical specification documents and approved makes list, and packaged into an RFQ. This flow is used both during tender cost estimation and post-award procurement — speed here directly determines how fast structured quotes come back.

The Design

Five-stage pipeline: chunk → interpret → extract → match → link. The key design choice is separating interpretation from extraction — the LLM reasons about what items mean first, then a second pass extracts clean structured data.

V1 shipped first with fixed 30-row chunks, processed sequentially. Each chunk’s last product block rolled forward as context to the next, so items split at boundaries weren’t lost. Recall was high — items were being extracted correctly — but the chunk size was a fixed parameter that didn’t adapt dynamically. Dense cable schedules got truncated, sparse civil sections wasted tokens, and sequential processing meant throughput couldn’t scale.

V2 replaced this with semantic chunking: the LLM identifies package boundaries based on the document’s own structure, not row counts. Since packages are self-contained, they can be processed in parallel. A scratchpad interpretation step was added between chunking and extraction — the model reasons about ambiguous items in free-form text first, then a second pass structures it. This separated reasoning from extraction, and enabled extracting the full BOQ hierarchy: packages → product groups → line items, with associated products and services separated from main items — not just a flat list of products.
V2 adds semantic chunking + scratchpad interpretation; V1 below for comparison

Key Decisions

Decision Why
Scratchpad before extraction Asking the LLM to interpret and extract in one pass conflates reasoning with structured output. A separate interpretation step lets the model reason about ambiguous descriptions in free-form text (higher temperature), then a second pass extracts clean structured data from that interpretation (lower temperature, strict schema). The temperature gradient matches the task — exploratory reasoning first, deterministic structuring second.
Semantic chunking (V1 → V2) V1’s fixed chunks worked — rolling context (forwarding the last product block) handled boundary splits. But chunk size didn’t adapt to content density, and sequential processing couldn’t scale. Semantic boundaries let the LLM split based on the document’s own structure, and since packages are self-contained, they can be processed in parallel.
Embed + LLM reranking Vector similarity against a 7k+ product catalog provides recall. An LLM reranking step applies domain-specific constraints — material must match, size must align — achieving precision without manual mapping rules. Matched products carry their vetted vendors from the database.
Semantic doc linking TDS spans from the tender summary index are classified into per-product sections. Each BOQ product is semantically matched to the right TDS section and approved makes entry. The LLM receives compact indices, not raw documents.

Results

8–10x faster — from BOQ upload to RFQ-ready output with matched vendors, specs, and approved makes
  • Handles BOQs with 1000+ line items across multiple sheets
  • In use by PSU and EPC clients on the Workwise platform
Module 3

Quotation Parsing (WIP)

After the BOQ-to-RFQ pipeline sends out vendor inquiries, quotes come back. Each vendor returns a PDF in their own format — different layouts, different line item descriptions, different ways of bundling products. The goal: extract structured product data from each quote and compare it against the original RFQ requirements to flag what’s matched, what deviates, and what’s missing.

Design Intent

The pipeline takes a vendor quote PDF and the corresponding RFQ as inputs. VLM/OCR extraction pulls structured product data from the quote — product names, specs, quantities, unit prices. Then a comparison step matches each extracted product against the RFQ line items and produces per-product flags:

  • Product matched — quoted product maps to an RFQ line item within spec
  • Deviation found — product matches but specs differ (e.g., different material grade, different rating)
  • Missing product — RFQ line item has no corresponding entry in the quote

The output feeds into cost estimation and vendor comparison workflows — procurement teams see a structured comparison rather than manually cross-referencing PDFs against spreadsheets.
