Module 1

Product & Vendor Database

The core challenge isn’t extraction — any LLM can pull specs from a product description. The hard part is deciding what “same product” means across sources that disagree on naming, attribute representation, and categorization — and building a catalog that gets smarter with every source it ingests rather than just bigger.

The Problem

Product data arrives from dozens of sources — vendor catalogs, quotations, historical BOQs — each with its own naming conventions, units, and attribute formats. “SS 316 Gate Valve 2 inch 150#” and “Stainless Steel Gate Valve 2″ Class 150” are the same product, but without normalization they appear as two entries. And both are one word away from “SS 316 Globe Valve 2 inch 150#” — which is a completely different product. When the same product exists as two catalog entries, vendor coverage fragments — vendors attached to one entry don’t appear when searching the other, so buyers see fewer options and get worse pricing.

Building this catalog manually was the status quo: data entry, cross-referencing, deduplication by eyeballing spreadsheets. At 7,000+ products and growing, with every new source adding variants, synonyms, and edge cases, manual curation doesn’t scale, leads to errors, and costs significant analyst time.

The Design

Three stages: ingest → review → dedup. Ingest processes each product one at a time — classify it, extract its specs, normalize to canonical forms, and write to the catalog. During ingestion, any attribute name or value the pipeline hasn’t seen before gets flagged. After all products are ingested, review resolves the flagged attributes in batch (approve as new, map to an existing synonym, or reject). Finally, dedup finds and merges duplicate variants across the catalog. The central idea: the catalog learns with every source it ingests — new naming conventions become synonyms, genuinely new attributes are absorbed, and duplicates are resolved without fragmenting vendor coverage.

Each product is ingested one at a time; flagged attributes and duplicates are resolved in batch after ingestion
Classify places each product into a 3-level taxonomy (e.g., Mechanical → Valves → Gate Valves). Classification determines the attribute set for extraction — valves get material/size/pressure class, cables get conductor count/voltage rating. A misclassified product gets the wrong attribute set and extracts the wrong specs, so classification gates everything downstream. The taxonomy grows organically: the LLM can propose new categories when existing ones don’t fit.

Extract + Normalize pulls the core product name and structured attributes, then maps each to canonical forms. Synonyms accumulate: “SS 316,” “Stainless Steel 316,” “SA 240 Gr 316” all resolve to the same canonical and skip the LLM entirely on future runs. When the pipeline encounters an attribute name or value it hasn’t seen before (e.g., a new spec like “surge_rating” or a new unit like “MVA”), it flags it for review rather than dropping it.

Review runs after all products are ingested. All flagged attributes for a category are reviewed together in one batch, so the LLM can spot overlaps (e.g., “pipe_material” and “material” are the same concept). Each flagged attribute is approved as a new canonical, mapped to an existing synonym, or rejected. Rejected values get a repair pass — reassigned or discarded, never silently lost.

Dedup runs in three sequential layers — progressively widening the comparison scope. A cheap similarity filter removes obvious non-matches before the LLM makes the final keep-or-merge decision on each ambiguous pair. The data model separates products (canonical groupings like “gate valve”) from variants (specific configurations like “gate valve, SS 316, 2″, 150#”), so merging variants doesn’t lose vendor associations — they transfer to the winning variant. Losers from each layer are excluded from the next to prevent cascading errors.

Key Decisions

Decision Why
Classify before extract Extraction quality depends on knowing the category. A valve and a cable need completely different attribute sets. Classification gates extraction — wrong category means wrong attributes means wrong dedup downstream.
Unknowns queue, not drop Attributes the pipeline hasn’t seen before don’t corrupt the catalog — they queue for batch review. The pipeline can be exploratory during ingestion because review is the gatekeeper. Approved unknowns become new canonicals or synonyms; rejected ones are excluded from future runs.
Jaccard + LLM dedup After ingestion, the catalog may have duplicate variants — same product described differently across sources. Comparing every pair via LLM is prohibitively expensive, so a cheap similarity score pre-filters to only ambiguous pairs, then the LLM decides keep or merge. Three sequential layers with loser exclusion prevent cascading merge errors.
Self-improving synonyms When the pipeline normalizes “SS 316” to “stainless steel 316,” that mapping is stored as a synonym. Next time any product has “SS 316,” it resolves instantly without an LLM call — the pipeline gets faster and more consistent with scale, not just bigger.

Results

7,000+ canonical products and growing — deduplicated, structured, with full spec attributes and vendor associations
  • New sources ingested without schema changes — upload CSV, run pipeline, catalog updates automatically
  • Catalog feeds BOQ-to-RFQ product matching, cost estimation, and vendor discovery across the Workwise platform
  • Deduplication consolidated vendor coverage per product — buyers see all available vendors in one place, not fragmented across duplicate entries
Module 2

Catalog Parsing (WIP)

When a new vendor is onboarded, their product catalog is the primary source of what they sell — but it arrives as a PDF. Tables span pages, cells merge unpredictably, headers repeat or don’t, and no two vendors format their catalogs the same way. Before the product database pipeline (Module 1) can classify and normalize these products, something has to turn the PDF into structured table data.

The Problem

Vendor product catalogs are dense PDF documents — product specs in tables that span multiple pages, merged cells, rotated headers, and embedded images mixed with text. Off-the-shelf PDF parsers fail on these layouts: they lose table structure, merge adjacent columns, or miss text inside complex cell arrangements. The product database can’t ingest what it can’t parse — and manual transcription from PDF catalogs is the bottleneck that limits how fast new vendor data enters the system.

The Design

Self-hosted PaddleOCR PPStructureV3, running CPU-only. The pipeline rasterizes each PDF page, then runs two parallel tracks: layout detection identifies regions (tables, text blocks, figures) and full-page OCR extracts all visible text. For each detected table region, a dedicated table structure recognition model identifies rows, columns, and cell boundaries, followed by cell-level OCR. The two OCR passes — full-page and table-specific — are merged via bounding box containment to catch text that table detection misses.

Layout detection splits the page; two OCR passes merge to produce structured table data
Why two-pass OCR? Table structure recognition identifies cell boundaries, and cell-level OCR extracts text within each cell. But some text falls inside the table’s bounding box yet outside any detected cell — merged headers, spanning captions, footnotes. The full-page OCR pass catches these. During consolidation, the two passes are merged via bounding box containment — text that the table model missed is recovered from the full-page pass rather than silently lost.

Key Decisions

Decision Why
CPU-only, self-hosted Catalog parsing runs on internal infrastructure without GPU dependency. No cloud OCR API costs, no data leaving the network, and the throughput is sufficient for batch catalog processing.
PaddleOCR PPStructureV3 Best open-source option for table structure recognition — handles merged cells, spanning headers, and multi-page tables that simpler OCR engines lose structure on.
Two-pass OCR Single-pass cell-level OCR loses text in merged headers, spanning captions, and footnotes — real data that the product database needs. A full-page OCR pass runs alongside and recovers what cell detection misses; the two are merged via bounding box containment. Text that falls inside a table region but outside any detected cell is tracked as “missing text” — surfacing it in a review/feedback loop is a next step.

Results

  • Vendor catalogs go from PDF to structured table data without manual transcription — directly feedable into the product database pipeline
  • Next steps: accuracy evaluation against ground truth, confidence-based routing for low-quality pages, and a review queue for flagged extractions
