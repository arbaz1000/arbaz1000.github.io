Flagship

FacctList — Sanctions Intelligence

The Problem

  • FacctList is Facctum's watchlist management platform — ingests, validates, and distributes sanctions lists to banks' screening systems
  • Regulators expect banks to act on sanctions as soon as press releases are published — but consolidated official lists can take significantly longer to follow. The gap is a compliance risk
  • Press releases come in mixed formats — sometimes tabular, sometimes free-form legal text — with entities, aliases, nationalities, and effective dates embedded inconsistently
  • Previously, analysts manually read press releases and updated the database — a process that took days per update

High-Level Design

  • Monitor polls EU press releases and official journals, detecting new sanctions publications
  • Extractor parses each publication and extracts structured records — entity names, aliases, nationalities, sanctions type, effective dates, legal basis
    • Format-aware parsing — different parsers handle tabular vs free-form legal text before a unified LLM extraction step
    • Publications span tens to hundreds of pages — reliable extraction at this scale is a core challenge
    • Alias extraction — free-form text lists full names, first/middle/last names, and aliases inconsistently; need to reliably extract all possible name combinations per entity. Evaluated prompt chaining vs fine-tuning, went with prompt chaining
  • Extracted records feed into FacctList's ingestion pipeline for deduplication, reconciliation, and distribution to downstream screening engines
Monitor → Extractor → FacctList ingestion pipeline

Results

96% — field-level extraction accuracy, evaluated over 5+ years of historical data
Days → minutes — press release to structured sanctions record
  • Integrated into FacctList's production pipeline, live with global clients
MVP

FacctGuard — Transaction Monitoring

The Problem

  • Banks need to flag suspicious transactions for AML at two levels:
    • Alert level — is this transaction suspicious given the customer's transaction history?
    • SAR level — should this transaction alert be escalated to a regulatory filing?

Approach

  • Built on 380M UK transactions using LSTM + XGBoost as complementary models:
    • LSTM — sequential patterns where transaction order matters (velocity, timing)
    • XGBoost — tabular features like amounts, counterparty risk, account aggregates
  • Model explainability layer — SHAP feature attributions translated into LLM-generated natural language summaries. Ops teams see “high-value transfer to a new counterparty in a high-risk jurisdiction” instead of feature weight vectors — actionable reasons, not raw model output

Results

3x lift in recall — at fixed review capacity vs rule-based alert generation. The ops team reviews the same number of alerts but catches 3x more truly suspicious cases
Infrastructure

Private LLM Inference Stack

The Problem

  • Compliance systems handle sensitive financial data — transactions, KYC documents, case files
  • Sending this to third-party LLM APIs is a regulatory and contractual non-starter for banking clients
  • But multiple systems depend on LLM capabilities — extraction, summarization, classification

Approach

  • Deployed self-hosted Qwen-2.5-14B with vLLM as the serving layer — full inference stack within Facctum's own infrastructure
  • Model selection — Qwen-2.5-14B with Apache 2.0 license (unlike Llama); outperforms Mistral at equivalent scale; fine-tunable architecture validated on regulatory document understanding tasks
  • vLLM provides continuous batching and PagedAttention — handles variable concurrent load efficiently without dedicating a GPU per request

Results

  • Removed the data residency blocker — Facctum can now serve regulated clients who would otherwise reject the product
